```markdown
# SKILL: Security Best Practices

## When to apply
Whenever code touches security-related logic.

## Rules
- Always sanitise session IDs before using them in file paths. re.sub(r'[^a-zA-Z0-9]', '', sid) prevents path traversal attacks (e.g. sid='../../etc/passwd').
- .claude/settings.local.json should never be committed to git — it contains local tool allowlists specific to one machine and is a minor info-leak if shared.
- Webhook HMAC verification must use hmac.compare_digest() (timing-safe). Both GitHub (x-hub-signature-256: sha256=<hex>) and Linear (Linear-Signature: <hex>) use HMAC-SHA256 but with different header formats.
- Pi-SEO scan detected 6 exposed keys in docs/runbooks and scripts/ files across dr-nrpg, synthex, ccw-crm (2026-04-10). Most are in documentation, not production code, but must still be verified as example values not real keys. Add pre-commit detect-secrets hook to every portfolio repo before Pi-SEO activation.
- When a Next.js API route (Vercel) uses process.env.ANTHROPIC_API_KEY directly (not via makeClient which trims), a trailing newline in the Vercel-stored key causes HTTP 401. Always call .trim() on env-sourced API keys before passing to the Anthropic SDK. This applies to any env var that goes into an HTTP Authorization header.
- When removing a hardcoded fallback secret (e.g. ADMIN_JWT_SECRET default value in code), the CI build step immediately fails because the variable was previously silently injected. Fix requires two steps: (1) add the secret to GitHub repo via 'gh secret set ADMIN_JWT_SECRET', (2) inject it in the workflow build step env block. Removing a fallback in code without updating CI secrets causes a hard CI failure on the next push.
```
