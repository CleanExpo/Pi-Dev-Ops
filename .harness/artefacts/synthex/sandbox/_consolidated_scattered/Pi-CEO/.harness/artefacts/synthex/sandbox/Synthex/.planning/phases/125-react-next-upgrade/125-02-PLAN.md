**Discovery findings (2026-03-20):**

**react-day-picker v8 → v9 (the major migration):**
`components/ui/calendar.tsx` uses v8 API:

- `classNames` with snake_case keys: `caption_label`, `nav_button_previous`, `nav_button_next`, `head_row`, `head_cell`, `day_selected`, `day_today`, `day_outside`, `day_disabled`, `day_range_middle`, `day_hidden`, `table`, `row`, `cell`
- `components.IconLeft` / `components.IconRight` pattern
- `showOutsideDays` prop

v9 renamed classNames keys (snake_case → new names) and changed the `components` prop:

- `components.IconLeft` / `components.IconRight` → replaced by `components.Chevron` (single component with `orientation` prop: `"left"` | `"right"`)
- classNames keys renamed (exact mapping: consult react-day-picker v9 types at `node_modules/react-day-picker/dist/index.d.ts` after install — the TypeScript errors from type-check will show the exact invalid keys)
- `showOutsideDays` → `hideOutsideDays={!value}` (inverted boolean, or prop may still work — check types)

**Strategy:** After npm install, run type-check. The TypeScript errors on calendar.tsx will show exactly which classNames keys are invalid in v9 — use these as the precise migration checklist. Consult `node_modules/react-day-picker/dist/index.d.ts` (ClassNames type) for the complete v9 key list.

Files using react-day-picker:

- `components/ui/calendar.tsx` — full API rewrite needed ✗
- `components/analytics/analytics-header.tsx` — type-only `import type { DateRange }` — likely fine (DateRange still exists in v9) ✓
- `app/dashboard/analytics/page.tsx` — type-only `import type { DateRange }` — likely fine ✓

**cmdk v0.2 → v1:**
`components/ui/command.tsx` wraps `CommandPrimitive` sub-components (`Input`, `List`, `Empty`, `Group`, `Separator`, `Item`) using `React.forwardRef` and `React.ComponentPropsWithoutRef<typeof CommandPrimitive.X>`. The cmdk v1 sub-component structure is the same — run type-check after install and fix only actual errors. Likely zero changes needed.

**@testing-library/react v14 → v16:**
Both test files already import `act` from `@testing-library/react` (not from `react-dom/test-utils`). No test file changes required — just the package bump resolves the React 19 peer incompatibility.

**Commit scope:** Stage ONLY the files actually changed in this phase. Use git add per-file — never `git add -A`.
