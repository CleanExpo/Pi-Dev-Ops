## Tasks

### Task 1: UI Contrast Audit

Scan all UI files for contrast and visibility issues. Work through the following categories:

**1a ‚Äî Input fields and labels:**

Scan files in `components/ui/`, `app/(auth)/`, `app/(onboarding)/`, `components/onboarding/`:

- Find all `<Input`, `<Textarea`, `<Select`, `<Checkbox`, `<RadioGroup` elements
- Check `className` for:
  - `text-white/[opacity]` where opacity ‚â§ 30 ‚Üí LOW CONTRAST
  - `placeholder:text-transparent` or `placeholder:text-white/10` ‚Üí INVISIBLE PLACEHOLDER
  - `border-transparent` on inputs with no background ‚Üí INVISIBLE BORDER
  - `bg-transparent` inputs on dark backgrounds with light text ‚Üí check carefully
  - Labels using `text-gray-500` or `text-gray-400` on `bg-surface-dark` ‚Üí may fail WCAG AA
- Also check for `<Label` components where className omits colour entirely (inherits invisible parent colour)

**1b ‚Äî Buttons and CTAs:**

Scan all `<Button` and `<button` elements across `components/`, `app/`:

- Ghost/outline variants: `variant="ghost"` or `variant="outline"` ‚Äî check they have a visible border
  or background on hover. `border-transparent` ghost buttons are invisible until hover.
- Disabled states: `disabled:opacity-50` is fine; `disabled:text-transparent` is not
- Icon-only buttons: verify `aria-label` and that the icon is visible (not `text-transparent`)

**1c ‚Äî Form panels and modals:**

Focus on `components/onboarding/`, `components/dashboard/`, any `*Dialog*`, `*Modal*`, `*Sheet*` files:

- Background: `bg-surface-base/80` with `backdrop-blur` ‚Äî check text on top uses sufficient contrast
- Error messages: look for `text-red-400` on `bg-red-500/10` ‚Äî may be low contrast in some browsers
- Success/info banners: `text-cyan-400` on `bg-cyan-500/5` ‚Äî very low contrast ratio, likely fails

**1d ‚Äî Addon / plugin / widget panels:**

Search for files containing "addon", "plugin", "widget", "integration" in their filename or directory:

```bash
find components/ app/ -name "*.tsx" | xargs grep -l "addon\|plugin\|widget\|integration" 2>/dev/null
```

For each found file, apply the same checks as 1b and 1c.

**1e ‚Äî Tailwind opacity pattern scan (automated):**

Run these greps to find patterns most likely to cause invisible text:

```bash
# Near-transparent text
grep -rn "text-white/[0-2][0-9]\b\|text-gray-[0-9]*/[0-2][0-9]\b" components/ app/ --include="*.tsx"

# Transparent placeholders
grep -rn "placeholder:text-transparent\|placeholder:text-white/[0-2][0-9]" components/ app/ --include="*.tsx"

# Transparent borders on interactive elements
grep -rn "border-transparent" components/ui/ components/onboarding/ app/ --include="*.tsx"

# bg-transparent inputs (check each manually)
grep -rn "bg-transparent" components/ui/ --include="*.tsx"

# text-gray-500 labels (borderline WCAG AA on dark bg)
grep -rn "text-gray-500" components/ui/ app/\(auth\)/ app/\(onboarding\)/ --include="*.tsx"
```

For each hit, read the surrounding context (5 lines) and classify:

- CRITICAL: text that is completely invisible (transparent or 0-opacity)
- HIGH: contrast ratio likely below 3:1 (fails large text AND normal text WCAG)
- MEDIUM: contrast ratio likely 3:1‚Äì4.5:1 (passes large text, fails normal text)
- LOW: borderline ‚Äî passes WCAG but user may struggle on certain screens

Document findings as:

```
[CONTRAST-xx] CRITICAL|HIGH|MEDIUM|LOW
File: <path>:<line>
Element: input | label | button | banner | placeholder | border
Class: <tailwind class causing issue>
Issue: invisible | low-contrast | missing-border
Fix: <suggested replacement class>
```
