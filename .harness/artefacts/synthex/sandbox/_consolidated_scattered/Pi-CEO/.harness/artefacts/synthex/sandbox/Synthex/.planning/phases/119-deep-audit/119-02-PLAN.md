# Plan 119-02: Route + Frontend Completeness Audit

**Phase:** 119 — Deep Audit
**Plan:** 02 of 03
**Linear:** Phase 119 (SYN issue TBC)
**Estimated scope:** Audit-only — no code changes, read and document findings

## Objective

Audit every API route for correctness and every frontend data-fetch for a live counterpart.
Find the gaps that cause silent 404s, 501 stubs, missing auth, and uncalled endpoints.

1. **Backend route audit** — scan `app/api/` for stubs, 501/500-only handlers, missing auth, missing Zod validation
2. **Frontend–backend connection audit** — find frontend `fetch()` / `useSWR()` calls that point to non-existent routes, and API routes that no frontend code calls

No code is written in this plan. Output is findings only.


Reference:

- CLAUDE.md § "Data Fetching Pattern" — three valid patterns: `useApi()`, `useSWR()`, server-side `fetch()`

- Zod rule: every POST/PUT/PATCH body must be parsed through a Zod schema before use

## Tasks

### Task 1: Backend Route Audit

Enumerate every file matching `app/api/**/**/route.ts`.

For each route file, check:

**1a — Stub detection:**
Search for any of these patterns that indicate an unimplemented handler:

- Body of handler contains only `return NextResponse.json({ error: 'Not implemented' })` or similar
- HTTP status 501
- `TODO` comment inside handler body
- Handler that immediately returns without any DB query or business logic

**1b — Auth guard:**
For every exported function (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`):

- Verify `getUserIdFromRequestOrCookies` (or `getServerSession`, or equivalent Supabase auth call) is present
- If auth call is absent for a mutation (POST/PUT/PATCH/DELETE) → CRITICAL
- If auth call is absent for a GET that returns user-scoped data → HIGH
- Public GET routes (e.g. `/api/health`, `/api/webhooks/`) → expected, note as OK

**1c — Zod validation:**
For every POST/PUT/PATCH handler:

- Search for `z.object(`, `z.string(`, `schema.parse(`, `schema.safeParse(`
- If none found and handler reads `request.json()` → HIGH (raw body usage)

**1d — Organization scoping:**
For every handler that queries Prisma:

- Verify the query includes `organizationId` or filters through a user→org relationship
- If query uses only `userId` without org scope on multi-tenant models → HIGH

Document findings as:

```
[ROUTE-xx] CRITICAL|HIGH|MEDIUM|LOW
File: app/api/<path>/route.ts:<line>
Handler: GET|POST|PUT|PATCH|DELETE
Issue: stub | missing-auth | missing-zod | missing-org-scope
Detail: <description>
```

### Task 2: Frontend–Backend Connection Audit

**2a — Collect all frontend API call targets:**

Search for `fetch(` and `useSWR(` in `app/`, `components/`, `hooks/` (client-side files only — files with `'use client'` or in hooks/):

```bash
grep -rn "fetch('/api/" app/ components/ hooks/ --include="*.tsx" --include="*.ts"
grep -rn "useSWR.*'/api/" app/ components/ hooks/ --include="*.tsx" --include="*.ts"
grep -rn "useApi.*'/api/" app/ components/ hooks/ --include="*.tsx" --include="*.ts"
```

Also capture `router.push('/api/...` and form `action="/api/..."` patterns.

For each `/api/<path>` found, verify a corresponding `app/api/<path>/route.ts` exists.
If route file is missing → HIGH (broken fetch — will 404 in production).

**2b — Collect all API routes and check for frontend callers:**

List all route files:

```bash
find app/api -name "route.ts" | sort
```

For each route, search whether any frontend file imports or calls that path string.
Routes with zero frontend callers → MEDIUM (may be called externally or via webhook — note uncertainty).
Routes prefixed `/api/webhooks/` or `/api/cron/` → exempt from this check (external callers).
Routes prefixed `/api/admin/` → check if admin UI calls them.

**2c — HTTP method mismatches:**

For each frontend `fetch('/api/path', { method: 'X' })`, verify the corresponding route.ts exports
the matching handler (`export async function X`). Method mismatch → HIGH.

Document findings as:

```
[CONNECT-xx] HIGH|MEDIUM|LOW
Frontend: <component path>:<line>
API: /api/<path>
Issue: missing-route | no-frontend-caller | method-mismatch
Detail: <description>
```
