## Classifier notes (consequential bug found during build)

The Board memo specifies `cancellation_details.reason = 'customer_requested'`
as the voluntary signal. **The actual Stripe API enum value is
`cancellation_requested`** (verified against the live cancellation event
that came back from the Unite-Group account today). The script handles
both values for robustness, but the memo's enum constant is wrong and
should be amended.

Stripe's documented enum (verified 2026-05-16):
- `cancellation_requested` — customer asked to cancel (voluntary)
- `payment_disputed` — chargeback (involuntary)
- `payment_failed` — dunning exhausted (involuntary)
- `null` — auto-cancelled without explicit reason (involuntary by default)

## Methodology (what the script will measure when the key is unlocked)

For each `customer.subscription.deleted` event in the window:

| classification | criteria                                                            |
|----------------|---------------------------------------------------------------------|
| voluntary      | `reason in ('cancellation_requested','customer_requested')` OR `feedback != null` |
| involuntary    | `reason in ('payment_failed','payment_disputed','unpaid')` OR (`reason IS NULL` AND `feedback IS NULL`) |

For each `customer.subscription.updated` event with `pause_collection != null`,
count as `paused` (proxy for stalled-payment customers heading to involuntary
churn).

Output JSON:
```json
{
  "voluntary": N,
  "involuntary": N,
  "paused": N,
  "total_cancellations": N,
  "feedback_top5": [["too_expensive", 42], ...],
  "reason_breakdown": {"cancellation_requested": 42, "payment_failed": 18, ...},
  "ratio": 0.43,
  "recommendation": "upgrade_flow_first | dunning_first | parallel",
  "mode": "live",
  "window_days": 30
}
```
