## The five whys

### Why #1 — Zero commits landed on `main` overnight.

Because the Pi-CEO autonomous build pipeline (Railway `linear_todo_poller` → `create_session` → Claude Agent SDK → git commit) never fired a single session on any of RA-597..604.

### Why #2 — Why did the poller never fire a session?

Because Railway production is running the PRE-MARATHON code. The seven commits that contain the self-start fix, the SDK Phase 3 migration, the bidirectional Telegram loop, and the charters are all **unpushed on local `main`** — they never left my sandbox. Railway only sees what's in `origin/main`, and `origin/main` hasn't moved since before the marathon.

### Why #3 — Why are the commits unpushed?

Because the Cowork sandbox has no SSH key and no authentication credentials for GitHub. I can `git commit` locally but I physically cannot `git push origin main` from here. The only path for these commits to reach Railway is for a human to run `git push origin main` from a machine that has push credentials — in practice, Phill's Mac.

Saturday night, my final Telegram message said "one `git push origin main` from your Mac ships all six unpushed commits." That wasn't a note — it was a **hard dependency on a manual human step** that I dressed up as a one-liner. When Phill went to bed without running it (perfectly reasonable, it was the end of a long day), the rails were doomed from that moment.

### Why #4 — Why did my plan depend on a manual push?

Because I designed the whole marathon around the assumption that "autonomous" = "Claude can run scheduled tasks inside Cowork" — when actually, the only truly always-on component in the entire stack is the Railway-hosted Pi-Dev-Ops FastAPI server. Everything else (scheduled tasks, watchdog, Telegram inbox, Pi-SEO dry-runs) runs **inside Cowork sandboxes that only exist when Cowork is open on Phill's Mac.** When Cowork isn't running, neither are the rails. When Cowork's sandbox rotates, the environment changes.

I built a "marathon system" that was really a "marathon on Phill's laptop while Cowork is open" system. That's not autonomous — it's a glorified scheduled background job in his IDE.

### Why #5 — Why did the watchdog escalate a CRITICAL alert about red tests when tests are actually green?

Because the watchdog runs `python3 -m pytest tests/` inside a fresh ephemeral Cowork sandbox every 30 minutes. Each sandbox is a clean slate. `pyproject.toml` declares `anthropic>=0.90` as a runtime dependency — but that dependency is NOT in the scheduled-task sandbox by default. When the import chain tries to load `app/server/sessions.py`, it blows up with `ModuleNotFoundError: No module named 'anthropic'`. Pytest exits 1. Watchdog classifies that as "tests red" and escalates a CRITICAL Telegram alert.

This is a **false-positive escalation pattern** that was sitting there from day one. The tests aren't red. The environment is broken. The watchdog can't tell the difference. So Phill gets woken up at 10:38 AEST on a Sunday with "CRITICAL: test suite red, need you to fix" — for a problem that doesn't exist.

A watchdog that cries wolf destroys its own usefulness. That's exactly what happened.

## Root cause (the single sentence)

**The autonomous rails weren't autonomous.** They were a set of scheduled tasks running inside ephemeral Cowork sandboxes that stopped when the Mac was asleep, read unpushed commits that never reached Railway, ran against unpredictable package environments, and escalated false-positive alerts to the founder instead of fixing anything.

The overnight failure wasn't one bug. It was the correct, expected outcome of the architecture I handed over Saturday night. It worked exactly as designed — and the design was wrong.

## What stopped the rails at 00:48 UTC Sunday

The scheduled-tasks MCP runs inside Cowork. When Cowork is not an active foreground process — because the Mac went to sleep, or the app was closed, or Phill logged out for the night — the scheduled-tasks scheduler can't wake up to fire its cron entries. The `lastRunAt` timestamps on every task stop at roughly the same moment: the last moment Cowork was alive.

Looking at the actual `lastRunAt` values:

- `marathon-telegram-inbox-5min`: 2026-04-12T00:48:33Z
- `marathon-watchdog-30min`: 2026-04-12T00:38:17Z
- `marathon-pi-seo-dryrun-hourly`: 2026-04-12T00:24:12Z
- `marathon-telegram-heartbeat`: 2026-04-11T23:03:14Z (fired ONCE, then never again)
- `pi-dev-ops-board-meeting`: 2026-04-11T20:02:47Z
- `linear-pi-dev-ops-sync`: 2026-04-11T23:03:13Z

They don't stop at the same instant, but they all stop before 01:00 UTC Sunday. That's roughly 09:00 AEST Sunday — almost exactly when a human might expect the laptop to have slept overnight and woken up in the morning.

## Recovery plan (four tracks, in priority order)

### Track 1 — Stop lying (today, 1 hour)
- **Silence the watchdog false-positive.** Add `anthropic` to the Cowork sandbox bootstrap for the watchdog task, OR gate the pytest check to `tests/` subdirectories that don't need the runtime deps, OR make the watchdog run `pytest --collect-only` which doesn't execute import chains that need runtime creds. Whichever is smallest.
- **Update every heartbeat and every escalation with a "running on: Cowork sandbox | Railway | other" field** so Phill can tell at a glance whether an alert came from the ephemeral rail or the real one.
- **Kill the marathon heartbeat if the Cowork rails are the only thing reporting**. A heartbeat that only fires when the laptop is awake is lying by omission.

### Track 2 — Move the autonomy to a real always-on place (this week)
- **Turn the autonomous loop into a Railway-hosted cron**. The Pi-Dev-Ops FastAPI server is already on Railway and running 24/7. Move the Pi-SEO dry-run, the watchdog, and the Linear sync into Railway background tasks (FastAPI `BackgroundTasks` + `asyncio.create_task` + an APScheduler) so they run whether Phill's Mac is open or not.
- **Set up a GitHub Actions workflow** (or Railway-native cron, whichever is cheaper) that runs pytest in a KNOWN environment with the full dependency set, on every commit and every 30 minutes on a timer. That becomes the real "is the test suite green" source of truth. The Cowork watchdog stops running pytest — it only reads a status file that GH Actions writes.
- **Give Pi-CEO a deploy token it can use to push.** Either a fine-grained GitHub PAT stored as a Railway secret and used by a `/api/admin/git-push` endpoint the autonomy poller can hit, OR a deploy key on the Pi-Dev-Ops Railway container that lets it push directly from the container's git. The goal: eliminate the human-in-the-loop push step that killed us overnight.

### Track 3 — Make /health tell the truth (this week)
- **The /health fix from commit e611b1c already includes `autonomy.armed` and `seconds_since_last_poll`.** Once it's actually deployed to Railway, the watchdog can read those fields and distinguish "poller is armed and working" from "poller is disabled" from "poller is dead." Until then, /health is the old version and still lies.
- **Add a self-test endpoint** `/api/admin/selftest` that runs a minimal end-to-end: fetch one Todo Linear issue, create an in-memory session against a tiny no-op repo, verify the SDK round-trips a response, tear it down, report result. The watchdog hits this every 30 minutes. If it comes back 200, the rails are truly working. If it comes back 503, we know where the break is.

### Track 4 — The scaling conversation (next week)
- **Reframe "autonomous across the portfolio" correctly.** We're not going to run a build session against CCW, RestoreAssist and DR-NRPG from inside the Pi-Dev-Ops container — the container doesn't have those repos mounted, doesn't have their secrets, doesn't have their Railway deploy hooks. We need a per-project Railway service (or a multi-project Pi-Dev-Ops with per-project `ANTHROPIC_API_KEY`, `GITHUB_TOKEN`, `REPO_URL` config) so the autonomous pipeline can scope its work to the right repo.
- **Start with the smallest, least-risky project** — probably Pi-Dev-Ops working on its own issues (which IS happening, it closed RA-585 autonomously this weekend — the ONE thing that worked). Once Pi-Dev-Ops fixing Pi-Dev-Ops is rock solid, clone the pattern to CCW.
